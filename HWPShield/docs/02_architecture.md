## 데이터 흐름 (Data Flow)

### 1. 업로드 단계

```
[사용자] ──▶ [드래그앤드롭] ──▶ [클라이언트 검증]
                                    │
                                    ▼ (magic bytes)
                              ┌────────────┐
                              │ .hwp 확인   │
                              │ D0 CF 11 E0 │
                              └─────┬──────┘
                                    │
                                    ▼ (50MB 제한)
                              ┌────────────┐
                              │ 크기 검증   │
                              └─────┬──────┘
                                    │
                                    ▼
                              [서버로 전송]
```

### 2. 분석 단계

```
[파일 수신] ──▶ [임시 저장] ──▶ [해시 계산]
                                    │
                                    ▼
                              [HWP 파싱]
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
               [스트림1]       [스트림2]       [스트림N]
                    │               │               │
                    └───────────────┼───────────────┘
                                    ▼
                         [병렬 모듈 분석]
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
              [탐지 결과1]    [탐지 결과2]    [탐지 결과N]
                    │               │               │
                    └───────────────┼───────────────┘
                                    ▼
                           [리스크 스코어 합산]
                                    │
                                    ▼
                              [JSON 리포트]
```

### 3. 응답 단계

```
[JSON 리포트] ──▶ [클라이언트 렌더링]
                      │
        ┌─────────────┼─────────────┐
        ▼             ▼             ▼
   [리스크 배지]  [점수 미터]   [모듈 결과]
        │             │             │
        └─────────────┴─────────────┘
                      │
                      ▼
```

## 보안 아키텍처

### 방어 계층

```
┌─────────────────────────────────────────┐
│  Layer 1: 클라이언트 검증                │
│  - 파일 확장자 (.hwp/.hwpx)             │
│  - Magic bytes (D0 CF 11 E0)           │
│  - 파일 크기 (50MB)                     │
└───────────────────┬─────────────────────┘
                    │
┌───────────────────▼─────────────────────┐
│  Layer 2: 전송 보안                      │
│  - HTTPS/TLS                            │
│  - 파일명 정제 (sanitization)           │
│  - MIME 타입 검증                      │
└───────────────────┬─────────────────────┘
                    │
┌───────────────────▼─────────────────────┐
│  Layer 3: 서버 처리                      │
│  - 임시 디렉토리 (isolated)              │
│  - Rate limiting (10 req/hour/IP)       │
│  - 타임아웃 (30초)                       │
└───────────────────┬─────────────────────┘
                    │
┌───────────────────▼─────────────────────┐
│  Layer 4: 정적 분석                      │
│  - 파일 실행 금지                        │
│  - 샌드박스 없음 (의도적)                │
│  - 스트림만 추출하여 검사                 │
└───────────────────┬─────────────────────┘
                    │
┌───────────────────▼─────────────────────┐
│  Layer 5: 정리                            │
│  - 분석 후 즉시 파일 삭제                 │
│  - 임시 디렉토리 스캔                    │
│  - 로그에서 민감 정보 마스킹              │
└─────────────────────────────────────────┘
```

## 확장성 고려사항

### 수평 확장

- 현재: 단일 인스턴스
- 향후: Redis 기반 Rate Limiting
- 향후: Celery + Redis 비동기 큐

### 성능 최적화

| 최적화 | 방법 | 효과 |
|--------|------|------|
| 스트림 캐싱 | zlib 해제 결과 메모리 캐시 | 30% 속도 향상 |
| 병렬 모듈 | asyncio.gather() | 40% 속도 향상 |
| 청크 업로드 | resumable upload | 대용량 파일 지원 |
| CDN | 정적 에셋 캐싱 | 로딩 시간 단축 |

## 배포 아키텍처

### Docker 구성

```yaml
# docker-compose.yml
services:
  frontend:
    build: ./frontend
    ports:
      - "3000:3000"
    environment:
      - REACT_APP_API_URL=http://localhost:8000
  
  backend:
    build: ./backend
    ports:
      - "8000:8000"
    environment:
      - MAX_FILE_SIZE=52428800
      - RATE_LIMIT=10

      - /tmp:size=100M,noexec,nosuid
```
